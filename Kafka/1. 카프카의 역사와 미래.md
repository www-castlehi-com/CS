## 1.1 탄생과 기본 구조
'링크드인'에서 파편화된 데이터 수집 및 분배 아키텍처를 운영하는 데에 큰 어려움을 겪었다.
데이터를 생성하고 적재하기 위해서는 데이터를 생성하는 소스 애플리케이션과 데이터가 최종 적재되는 타깃 애플리케이션을 연결해야 한다.
초기 운영 시에는 단방향 통신을 통했기에 아키텍처가 복잡하지 않았으므로 운영이 힘들지 않았는데, 시간이 지날수록 아키텍처는 거대해졌고 소스 애플리케이션과 타깃 애플리케이션의 개수가 점점 많아지면서 데이터를 전송하는 라인이 복잡해졌다.
![](https://i.imgur.com/BJzf25q.png)
이를 해결하기 위해 각종 상용 데이터 프레임워크와 오픈소스를 아키텍처에 녹여내어 데이터 파프라인의 파편화를 개선하려고 했지만 복잡도를 낮출 수 없었다.
![](https://i.imgur.com/YEQxfWQ.png)
결국 신규 시스템을 독자적으로 만들기로 결정했고, 그 결과물이 아파치 카프카다. 
카프카는 각각의 애플리케이션이 연결하여 데이터를 처리하는 것이 아니라 한 곳에 모아 처리할 수 있도록 중앙집중화했다.
![](https://i.imgur.com/w0TBpD1.png)
소스 애플리케이션은 어느 타깃 애플리케이션으로 보낼 것인지 고민하지 않고 카프카에게 넣는다.
카프카 내부에 데이터가 저장되는 파티션의 동작은 FIFO 방식의 큐 자료구조와 유사하다.
큐에 데이터를 보내는 것이 '프로듀서'이고, 큐에서 데이터를 가져가는 것이 '컨슈머'이다.
## 1.2 데이터파이프라인으로 적합한 이유
### 1.2.1 높은 처리량
카프카는 아래의 경우를 모두 배치 처리를 한다.
- 프로듀서가 브로커로 데이터 송신
- 컨슈머가 브로커로부터 데이터 수신
배치 처리를 할 경우 동일한 양의 데이터를 보낼 때 네트워크 통신 횟수를 최소한으로 줄일 수 있기 때문에 동일 시간 내에 더 많은 데이터를 전송할 수 있다.

하나의 토픽에 대하여 파티션 단위를 통해 동일 목적의 데이터를 여러 파티션에 분배하고 데이터를 병렬 처리할 수 있다.
파티션 개수만큼 컨슈머 개수를 늘려서 동일 시간당 데이터 처리량을 늘릴 수 있다.
### 1.2.2 확장성
트래픽이 갑자기 몰릴 때 안정적으로 확장이 가능하다.
카프카는 브로커를 클러스터로 운영한다. 프로듀서가 메시지를 브로커로 보내면 브로커는 해당 메시지를 보관한다.
데이터가 적을 때에는 카프카 클러스터의 브로커를 최소한의 개수로 운영하다가 데이터가 많아지면 클러스터의 브로커 개수를 자연스럽게 늘려 스케일 아웃할 수 있다. 반대로 데이터가 적어지고 추가 서버들이 더 필요 없어지면 브로커 개수를 줄여 스케일 인할 수 있다.
스케일 아웃, 스케일 인은 무중단 운영을 지원하므로 365일 24시간 데이터를 처리해야 하는 커머스나 은행 같은 비즈니스 모델에서도 안정적인 운영이 가능하다.
### 1.2.3 영속성
카프카는 전송받은 데이터를 메모리에 저장하지 않고 파일 시스템에 저장하기 때문에 프로그램이 종료되더라도 데이터가 사라지지 않는다.

파일 시스템을 사용하는 것은 보편적으로 느리지만 카프카는 운영체제 레벨에서 파일시스템을 최대한 활용하는 방법을 적용했다.
운영체제에서는 파일 I/O 성능 향상을 위해 페이지 캐시 영역을 메모리에 따로 생성하여 사용한다.
페이지 캐시 메모리 영역을 사용하여 한 번 읽은 파일 내용은 메모리에 저장시켰다가 다시 사용하는 방식이어서 파일 시스템에 저장한 후 데이터를 저장, 전송하더라도 처리량이 높다.

위처럼 디스크 기반의 파일 시스템을 활용한 덕분에 브로커 프로세스 장애가 발생하더라도 재시작하고 안전하게 데이터를 다시 처리할 수 있다.
### 1.2.4 고가용성
카프카 클러스터는 최소 3대의 브로커로 구성되어 있다.
프로듀서로부터 받은 메시지를 복제를 통해 모든 브로커가 저장을 한다.
한 브로커에 장애가 발생하더라도 복제된 데이터가 나머지 브로커에 저장되어 있으므로 지속적으로 데이터 처리가 가능하다.
따라서 일부 서버에 장애가 발생하더라도 무중단으로 안전하고 지속적으로 데이터를 처리할 수 있다.

또한 온프레미스 환경의 서버 랙 또는 퍼블릭 클라우드의 리전 단위 장애에도 데이터를 안전하게 복제할 수 있는 브로커 옵션들이 있다.
## 1.3 빅데이터 아키텍처의 종류와 카프카의 미래
### 1.3.1 초기 빅데이터 플랫폼
![](https://i.imgur.com/keGn8Rp.png)
초기 빅데이터 플랫폼은 엔드 투 엔드로 각 서비스 애플리케이션으로부터 데이터를 배치로 모았다.
### 1.3.2 람다 아키텍처
![](https://i.imgur.com/nNDkoXB.png)
람다 아키텍처는 배치 레이어, 스피드 레이어, 서빙 레이어로 나뉜다.
배치 레이어는 배치 데이터를 모아서 특정 시간, 타이밍마다 일괄 처리한다.
스피드 레이어는 서비스에서 생성되는 원천 데이터를 실시간으로 분석하는 용도로 사용하며 카프카가 위치한다.
서빙 레이어는 배치 레이어, 스피드 레이어에서 가공된 데이터를 데이터 사용자, 서비스 애플리케이션이 사용할 수 있도록 데이터를 저장한다.

하지만 데이터를 처리하는 레이어가 2개로 나뉘기 때문에 배치 데이터와 실시간 데이터를 융합하여 처리할 때는 유연하지 못한 파이프라인을 생성해야 한다.
1개의 로직을 추상화하여 배치, 스피드 레이어에 적용하는 서밍버드가 있었지만 디버깅 시 각각을 디버깅해야하는 문제는 여전했다.
### 1.3.3 카파 아키텍처
![](https://i.imgur.com/RE8hPYV.png)
람다 아키텍처의 단점으로 부각되었던 로직의 파편화, 디버깅, 배포, 운영 분리에 대한 이슈를 제거하기 위해 배치 레이어를 제거했다.
배치 레이어에서 하던 작업도 스피드 레이어에서 처리하는 아키텍처이다.
![](https://i.imgur.com/XEeabJ8.png)
로그는 배치 데이터를 스트림으로 표현하기에 적합하다. 
배치 데이터를 로그로 표현할 때는 각 시점의 배치 데이터의 변환 기록을 시간 순서대로 기록하여 각 시점의 모든 스냅샷 데이터를 저장하지 않고도 배치 데이터를 표현할 수 있게 되었다.

배치데이터는 한정된 데이터를 처리하고 대규모 데이터를 위한 분산 처리를 수행한다.
따라서 분, 시간, 일 단위 처리를 위한 지연이 발생한다.
대신 대규모 데이터를 들고 있기에 복잡한 키 조인 수행이 가능하다.

스트림 데이터는 무한한 데이터 처리가 가능하며 지속적으로 들어오는 데이터를 위한 분산 처리를 수행한다.
따라서 분 단위 이하의 지연이 발생한다.
짧은 시간 구간 안의 데이터끼리만 엮으므로 단순한 키 조인 수행이 가능하다.
![](https://i.imgur.com/SXRSHJ7.png)
카프카는 로그에 시간(timestamp)를 남겨 닫힌 구간으로 만들어 배치 처리를 가능하게 만든다.
kafka streams application은 데이터를 materialized view로 가져와 처리한다.
### 1.3.4 스트리밍 데이터 레이크
![](https://i.imgur.com/rO5qkMA.png)
카프카에 분석과 프로세싱을 완료한 거대한 용량의 데이터를 오랜 기간 저장하고 사용할 수 있다면 서빙 레이어를 제거하고 스피드 레이어만 관리할 수 있다.
현재는 어렵지만 kafka도 계속해서 개선점을 찾아가고 있다.
![](https://i.imgur.com/VCd5DiZ.png)
자주 접근하지 않는 데이터는 비싼 자원(브로커의 메모리, 디스크)에 유지하지 않고 오브젝트 스토리지와 같은 저렴하면서 안전한 저장소에 옮겨 저장한다.
자주 사용하는 데이터만 브로커에서 사용하여 구분한다.
현재는 제공되고 있지 않다.
