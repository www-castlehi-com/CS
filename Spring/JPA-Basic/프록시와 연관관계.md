## 1. 프록시
### 프록시란?
프록시란 가짜 객체를 의미한다.

JPA에서는 `em.find()`외에도 `em.getReference()`를 통해 데이터베이스에서 데이터를 조회할 수 있다.
`em.find()`는 데이터베이스를 통해 실제 엔티티 객체를 조회하지만,
`em.getReference()`는 데이터베이스 조회를 미루는 가짜(프록시) 엔티티 객체를 조회한다.

```java
Member findMember = em.find(Member.class, member.getId());  
```
```sql
select
	m1_0.MEMBER_ID,
	m1_0.createdBy,
	m1_0.createdDate,
	m1_0.lastModifiedBy,
	m1_0.lastModifiedDate,
	l1_0.LOCKER_ID,
	l1_0.name,
	t1_0.TEAM_ID,
	t1_0.name,
	m1_0.USERNAME 
from
	Member m1_0 
left join
	Locker l1_0 
		on l1_0.LOCKER_ID=m1_0.LOCKER_ID 
left join
	Team t1_0 
		on t1_0.TEAM_ID=m1_0.TEAM_ID 
where
	m1_0.MEMBER_ID=?
```
`em.find()`의 경우 select 쿼리가 발생한다.

```java
Member findMember = em.getReference(Member.class, member.getId());
```

`em.getRefrence()`의 경우 select 쿼리가 발생하지 않았다.
위의 java 예시는 조회 전 데이터를 삽입하는 과정이 포함되어 있는데, 생략되었다.
sql에서는 데이터를 삽입하는 sql만 발생하고 조회 쿼리가 발생하지 않는다.
```java
Member findMember = em.getReference(Member.class, member.getId());  
System.out.println("findMember.getId() = " + findMember.getId());  
System.out.println("findMember.getUsername() = " + findMember.getUsername());
```

즉, 가짜 객체를 가져왔다가 값을 실제로 사용할 때 select 쿼리가 발생한다.

이 가짜 객체가 **Proxy** 임을 알 수 있다.
### 특징
![](https://i.imgur.com/IZBtsVg.png)
프록시는 실제 클래스를 상속 받아서 만들어지기 때문에 실제 클래스와 겉 모양이 같다.
![](https://i.imgur.com/5JyGlop.png)
프록시 객체는 실제 객체의 참조 (target)를 보관한다. 초기에 이는 null로 초기화되어있어 id만 가진 껍데기 객체일 뿐이다.
프록시 객체를 호출하면 프록시 객체는 실제 객체의 메서드를 호출한다.
![](https://i.imgur.com/4TVDfRu.png)
프록시 객체의 메서드를 호출할 경우, 프록시 객체는 JPA 영속성 컨텍스트에게 초기화를 요청한다.
그러면 DB에서 조회에서 나온 실제 Entity의 실제 메서드가 호출된다.

프록시 객체는 처음 사용할 때 한 번만 초기화된다.
초기화 시 프록시 객체가 실제 엔티티로 바뀌는 것은 아니며 접근 가능한 객체가 될 뿐이다.

만약, 프록시 객체를 초기화할 때 영속성 컨텍스트에 엔티티가 이미 있으면 어떻게 될까?
```java
Member member1 = em.find(Member.class, member.getId());  
System.out.println("member1.getClass() = " + member1.getClass());  
Member reference = em.getReference(Member.class, member.getId());  
System.out.println("reference.getClass() = " + reference.getClass());
```
```log
member1.getClass() = class hellojpa.Member
reference.getClass() = class hellojpa.Member
```
프록시 객체를 반환하지 않고, 실제 객체를 반환한다.

또 프록시 객체를 초기화하는 시점에 영속성 컨텍스트의 도움을 받을 수 없는 준영속 상태일 경우에 프록시를 초기화하면 어떻게 될까?

### 확인
#### 1) 초기화 여부
```java
emf.getPersistenceUtil().isLoaded(Object entity)
```
프록시가 초기화 되어 있는지 여부를 확인한다.

```java
System.out.println("emf.getPersistenceUnitUtil().isLoaded(reference) = " + emf.getPersistenceUnitUtil().isLoaded(reference));  
reference.getUsername();  
System.out.println("emf.getPersistenceUnitUtil().isLoaded(reference) = " + emf.getPersistenceUnitUtil().isLoaded(reference));
```
이 경우 첫 출력문에는 false, 그 다음 출력문에는 true가 출력된다.
#### 2) 클래스 확인
```java
entity.getClass()
```
프록시 객체가 어떤 클래스를 상속하는지 확인할 수 있다.
```log
findMember.getClass() = class hellojpa.Member$HibernateProxy$Ef3it5jJ
```
#### 3) 강제 초기화
```java
org.hibernate.Hibernate.initialize(entity);
```
JPA 표준은 강제 초기화 방법이 없다.
보통 프록시의 메서드를 호출해서 초기화를 하곤 한다.

hibernate에서는 강제 초기화를 지원한다.
위와 같이 entity를 강제 초기화 할 경우, select 쿼리가 발생한다.
## 2. 로딩
### 1️⃣ 즉시 로딩
엔티티를 조회할 때 연관된 엔티티도 함께 조회한다.
`@ManyToOne`과 `@OneToOne`가 사용하는 기본 fetch type이다.
```java
@Entity  
public class Member extends BaseEntity {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
  
    @ManyToOne(fetch = FetchType.EAGER)  
    @JoinColumn(name = "TEAM_ID")  
    private Team team;
}
```
`@ManyToOne`의 fetch 속성을 `FetchType.EAGER`로 지정한다.
```sql
select
	m1_0.MEMBER_ID,
	m1_0.createdBy,
	m1_0.createdDate,
	m1_0.lastModifiedBy,
	m1_0.lastModifiedDate,
	t1_0.TEAM_ID,
	t1_0.name,
	m1_0.USERNAME 
from
	Member m1_0 
left join
	Team t1_0 
		on t1_0.TEAM_ID=m1_0.TEAM_ID 
where
	m1_0.MEMBER_ID=?
```
member 조회 시 team을 join하여 함께 데이터를 가져오는 sql이 발생한다.
member를 조회하려고 했지만 연관된 모든 객체가 반환되어 예상치 못한 SQL이 발생한다.

```java
List<Member> members = em.createQuery("select m from Member m", Member.class).getResultList();
```
`em.find()` 대신 JPQL을 사용해보자.
```sql
Hibernate: 
select
		m1_0.MEMBER_ID,
		m1_0.createdBy,
		m1_0.createdDate,
		m1_0.lastModifiedBy,
		m1_0.lastModifiedDate,
		m1_0.LOCKER_ID,
		m1_0.TEAM_ID,
		m1_0.USERNAME 
	from
		Member m1_0
Hibernate: 
select
	t1_0.TEAM_ID,
	t1_0.name 
from
	Team t1_0 
where
	t1_0.TEAM_ID=?
```
select 쿼리가 예상과 다르게 2개가 발생했다.
`em.find()`는 조회 로직이 최적화 되어 있지만 JPQL은 PK로 조회를 하며 작성한 query를 그대로 sql 문법으로 변경한다.
그래서 member를 조회하고 FK를 이용해 team을 또 조회하게 된다.
결국, N+1 문제가 발생한다.
### 2️⃣ 지연 로딩
연관된 엔티티를 실제 사용할 때 조회한다.
`@OneToMany`, `@ManyToMany`가 사용하는 기본 fetch type이다. 
> ToMany 류의 연관관계의 경우, 대량의 데이터를 조회할 수 있는 가능성이 있어 이를 방지하기 위해 지연 로딩을 기본으로 사용한다.
```java
@Entity  
public class Member extends BaseEntity {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
  
    @ManyToOne(fetch = FetchType.LAZY)  
    @JoinColumn(name = "TEAM_ID")  
    private Team team;
}
```
`@ManyToOne`의 fetch 속성을 `FetchType.LAZY`로 지정한다.
```java
Team team = new Team();  
team.setName("teamA");  
em.persist(team);  
  
Member member = new Member();  
member.setUsername("hello");  
member.setTeam(team);  
em.persist(member);  
  
em.flush();  
em.clear();  
  
Member m = em.find(Member.class, member.getId());  
System.out.println("m.getTeam().getClass() = " + m.getTeam().getClass());
```
member, team을 연관지어 저장하고 member를 조회했다.
```sql
select
	m1_0.MEMBER_ID,
	m1_0.createdBy,
	m1_0.createdDate,
	m1_0.lastModifiedBy,
	m1_0.lastModifiedDate,
	m1_0.TEAM_ID,
	m1_0.USERNAME 
from
	Member m1_0 
where
	m1_0.MEMBER_ID=?

m.getTeam().getClass() = class hellojpa.Team$HibernateProxy$jICYXdbz
```
Member를 조회한 sql에서는 team을 조인하지 않았고, 조회된 member로부터 가져온 team은 proxy 객체임을 알 수 있다.

```java
m.getTeam().getName();
```
```sql
select
	t1_0.TEAM_ID,
	t1_0.name 
from
	Team t1_0 
where
	t1_0.TEAM_ID=?
```
그리고 team의 메서드 호출 시 team 조회 sql이 발생한다.
