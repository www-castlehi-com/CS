관계형 데이터베이스는 상속 관계를 표현할 방법이 없지만, 슈퍼타입과 서브타입 관계라는 모델링 기법을 유사하게 사용할 수 있다.

기본적으로 객체의 상속 관계는 데이터베이스에서 어떻게 표현될까?
아래와 같이 객체를 상속 구조로 생성한다.
```java
@Entity  
public class Item {  
  
    @Id  
    @GeneratedValue    
    private Long id;  
  
    private String name;  
  
    private int price;  
}

@Entity  
public class Album extends Item {  
  
    private String artist;  
}

@Entity  
public class Movie extends Item {  
  
    private String director;  
    private String actor;
}

@Entity  
public class Book extends Item {  
  
    private String author;  
    private String isbn;  
}
```
생성되는 SQL은 다음과 같다.
```sql
create table Item (
	price integer not null,
	id bigint not null,
	DTYPE varchar(31) not null,
	actor varchar(255),
	artist varchar(255),
	author varchar(255),
	director varchar(255),
	isbn varchar(255),
	name varchar(255),
	primary key (id)
)
```
기본값은 하나의 테이블에 모든 상속 객체의 필드들이 포함됨을 알 수 있다.

이는 상속 관계가 표현된 것이 아니다.
그렇다면 데이터베이스에서 상속 관계를 표현하는 방법은 무엇이 있을까?
## 1. 조인 전략
```java
@Entity  
@Inheritance(strategy = InheritanceType.JOINED)  
public class Item {  
  
    //...
}
```
기본값 예시에서 부모 클래스에 `@Inheritance(strategy = InheritanceType.JOINED)`를 지정한다.

생성되는 SQL의 결과는 아래와 같다.
```sql
create table Item (
	price integer not null,
	id bigint not null,
	name varchar(255),
	primary key (id)
)

create table Album (
	id bigint not null,
	artist varchar(255),
	primary key (id)
)

create table Book (
	id bigint not null,
	author varchar(255),
	isbn varchar(255),
	primary key (id)
)

create table Movie (
	id bigint not null,
	actor varchar(255),
	director varchar(255),
	primary key (id)
)
```
객체로 상속 관계를 나타냈던 것처럼 테이블이 각 클래스에 맞게 생성되었다.

이제 데이터로 확인 해보자.
```java
Movie movie = new Movie();  
movie.setDirector("aaaa");  
movie.setActor("bbbb");  
movie.setName("바람과함께사라지다");  
movie.setPrice(10000);  

em.persist(movie);
tx.commit();
```
![](https://i.imgur.com/Z5GfmIl.png)
부모 클래스를 나타내는 테이블인 Item 뿐만 아니라 자식 클래스를 나타내는 테이블인 Movie에도 정상적으로 데이터 삽입이 되었다.

```java
Movie findMovie = em.find(Movie.class, movie.getId());
```
```sql
select
	m1_0.id,
	m1_1.name,
	m1_1.price,
	m1_0.actor,
	m1_0.director 
from
	Movie m1_0 
join
	Item m1_1 
		on m1_0.id=m1_1.id 
where
	m1_0.id=?
```
조회 시에는 부모 테이블인 Item과 자식 테이블인 Movie를 join 한다.

그런데 Item만 조회하는 경우, 한 가지 불편함이 있다.
조회하려는 데이터가 어떤 자식 타입인지 알기 어렵다는 것이다.
이를 알아내기 위해서는 자식 테이블을 모두 join 해 봐야 한다.
그래서 JPA는 이를 구분할 수 있는 어노테이션을 지원한다.
```java
@Entity  
@Inheritance(strategy = InheritanceType.JOINED)  
@DiscriminatorColumn(name = "TYPE")
public class Item {
}

@Entity  
@DiscriminatorValue("A")  
public class Album extends Item {
}

@Entity  
@DiscriminatorValue("B")  
public class Book extends Item {
}

@Entity  
@DiscriminatorValue("M")  
public class Movie extends Item {
}
```
`@DiscriminatorColumn`
- 부모 클래스에 지정한다.
- 테이블에 저장된 각 row가 어떤 엔티티 타입인지 구분하기 위한 컬럼이다.
- name을 지정하지 않을 경우, "DTYPE"으로 기록된다.
`@DiscriminatorValue`
- 자식 클래스에 지정한다.
- 해당 엔티티가 discriminator 컬럼에 저장될 실제 값이다.

결과적으로 아래와 같이 조인 없이도 어떤 자식 테이블의 데이터인지 확인할 수 있다.
![](https://i.imgur.com/Png7PJ9.png)

상속 관계 매핑이 비즈니스 적으로 중요한 의미를 가지고 있을 경우 사용한다.
## 2. 단일 테이블 전략
```java
@Entity  
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)  
public class Item {
}
```
InheritanceType을 SINGLE_TABLE로 바꾼다.
```sql
create table Item (
	price integer not null,
	id bigint not null,
	DTYPE varchar(31) not null,
	actor varchar(255),
	artist varchar(255),
	author varchar(255),
	director varchar(255),
	isbn varchar(255),
	name varchar(255),
	primary key (id)
)
```
모든 자식 클래스의 필드도 부모 클래스의 필드에 포함된다.
`@Inheritance`를 설정하지 않았을 경우와 생김새가 유사하다.

모든 자식 데이터도 함께 쌓이게 되므로 구분을 위해 `@DiscriminatorColumn` 설정이 존재하지 않아도 자동으로 추가된다.

상속 관계 매핑이 굳이 필요하지 않으며, 테이블의 변경이 잦지 않을 경우 성능을 고려하여 많이 사용한다.
## 3. 구현 클래스마다 테이블 전략
```java
@Entity  
@Inheritance(strategy = InheritanceType.TABLE_PER_CLASS)  
@DiscriminatorColumn(name = "TYPE")  
public abstract class Item {
}
```
InheritanceType을 TABLE_PER_CLASS로 바꾼다.
또 명확한 상속을 표현하기 위해 abstract class로 변경했다.
다른 전략 또한 명확히 상속 구조를 표현하기 위해서는 abstract class로 생성하는 것이 좋다.
```sql
create table Album (
	price integer not null,
	id bigint not null,
	artist varchar(255),
	name varchar(255),
	primary key (id)
)

create table Book (
	price integer not null,
	id bigint not null,
	author varchar(255),
	isbn varchar(255),
	name varchar(255),
	primary key (id)
)

create table Movie (
	price integer not null,
	id bigint not null,
	actor varchar(255),
	director varchar(255),
	name varchar(255),
	primary key (id)
)
```
Item 테이블은 생성되지 않고, Item 테이블의 컬럼들이 모두 각 자식 테이블의 컬럼에 구현되었다.

삽입 시에는 자식 테이블에 모두 데이터가 삽입된다.
![](https://i.imgur.com/PNREuqI.png)
조회 시에는 join할 필요 없이 대상 테이블에서 insert 쿼리가 실행된다.
```sql
select
	m1_0.id,
	m1_0.name,
	m1_0.price,
	m1_0.actor,
	m1_0.director 
from
	Movie m1_0 
where
	m1_0.id=?
```
하지만, 부모 테이블을 기준으로 조회를 할 시에는 모든 자식테이블을 확인해야 한다.
```sql
select
	i1_0.id,
	i1_0.clazz_,
	i1_0.name,
	i1_0.price,
	i1_0.artist,
	i1_0.author,
	i1_0.isbn,
	i1_0.actor,
	i1_0.director 
from
	(select
		price,
		id,
		artist,
		name,
		null as author,
		null as isbn,
		null as actor,
		null as director,
		1 as clazz_ 
	from
		Album 
	union
	all select
		price,
		id,
		null as artist,
		name,
		author,
		isbn,
		null as actor,
		null as director,
		2 as clazz_ 
	from
		Book 
	union
	all select
		price,
		id,
		null as artist,
		name,
		null as author,
		null as isbn,
		actor,
		director,
		3 as clazz_ 
	from
		Movie
) i1_0 
where
    i1_0.id=?
```
UNION SQL로 인해 성능 저하뿐만 아니라 자식 테이블을 통합해서 쿼리하기 어렵기 때문에 제일 사용하지 않는 상속 관계 매핑 전략이다.
