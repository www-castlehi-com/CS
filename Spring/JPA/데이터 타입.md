## 1️. 엔티티 타입
`@Entity`로 정의하는 객체이다.
데이터가 변해도 식별자로 지속해서 추적이 가능하다.
_ex. 회원 엔티티의 키나 나이 값을 변경해도 식별자로 인식 가능_
## 2. 값 타입
단순히 값으로 사용하는 자바 기본 타입이나 객체를 의미한다.
식별자가 없고 값만 있으므로 변경 시 추적이 불가능하다.
_ex. 숫자 100을 200으로 변경하면 완전히 다른 값으로 대체_
### 1️⃣ 기본값 타입
_ex. String name, int age_
회원을 삭제하면 이름, 나이 필드도 함께 삭제되므로 생명주기가 엔티티에 의존한다.
회원 이름 변경 시 다른 회원의 이름도 함께 변경되면 안되므로 다른 객체와 공유해서는 안 된다.

> `int`, `double` 같은 기본 타입은 공유가 되지 않는다.
> 위와 같은 기본 타입은 항상 값을 복사한다.
> `Integer`같은 래퍼 클래스나 `String`같은 특수한 클래스는 공유가 가능하지만 값 자체를 변경할 수 있는 기능을 지원하지 않기 때문에 안전하게 사용할 수 있다.
### 2️⃣ 임베디드 타입
새로운 값 타입을 직접 정의할 수 있다.
#### 2.1 사용
```java
@Embeddable  
public class Address {  
  
    private String city;  
  
    private String street;  
  
    private String zipcode;
}
```
위와 같이 임베디드 타입 클래스를 생성했다고 해보자.
`@Embeddable`은 값 타입을 정의하는 곳에 표시한다.
```java
@Entity  
public class Member extends BaseEntity {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
  
    @Embedded  
    private Address homeAddress;
}
```
그리고 이 임베디드 타입을 위와 같이 사용하고 있다고 해보자.
`@Embedded`는 값 타입을 사용하는 곳에 표시한다.

그렇다면 SQL은 아래와 같이 수행된다.
```sql
create table Member (
	MEMBER_ID bigint not null,
	USERNAME varchar(255),
	city varchar(255),
	street varchar(255),
	zipcode varchar(255),
	primary key (MEMBER_ID)
)
```
클래스는 분리했지만, 테이블 상에서는 같은 테이블에 위치해 있는 것을 볼 수 있다.
#### 2.2 장점
클래스 분리가 되므로 재사용이 가능하고 높은 응집성을 가진다.
#### 2.3 연관관계
![](https://i.imgur.com/KdOTf0q.png)
임베디드 타입은 여러 value 타입을 가질 수 있고, 여러 entity도 가질 수 있다.
임베디드 타입이 FK를 가지고 있어도 테이블 상 문제가 없다.
#### 2.4 속성 재정의
한 엔티티에서 같은 임베디드 값 타입을 여러 개 사용할 경우 컬럼 명이 중복된다.

```java
@Entity  
public class Member extends BaseEntity {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
  
    @Embedded  
    private Address homeAddress;  
  
    @Embedded  
    @AttributeOverrides({  
          @AttributeOverride(name = "city", column = @Column(name = "WORK_CITY")),  
          @AttributeOverride(name = "street", column = @Column(name = "WORK_STREET")),  
          @AttributeOverride(name = "zipcode", column = @Column(name = "WORK_ZIPCODE"))  
    })  
    private Address workAddress;
}
```
```sql
create table Member (
	MEMBER_ID bigint not null,
	USERNAME varchar(255),
	WORK_CITY varchar(255),
	WORK_STREET varchar(255),
	WORK_ZIPCODE varchar(255),
	city varchar(255),
	street varchar(255),
	zipcode varchar(255),
	primary key (MEMBER_ID)
)
```
이 경우 `@AttributeOverrides`를 사용해서 컬럼 명 속성을 재정의 한다.
#### 2.5 null값
임베디드 타입의 값이 null이면 매핑한 컬럼 값도 모두 null로 입력된다.
#### 2.6 주의
여러 엔티티에서 공유할 경우 side effect가 발생한다.
```java
Address address = new Address("city", "street", "10000");  
  
Member member = new Member();  
member.setUsername("member1");  
member.setHomeAddress(address);  
em.persist(member);  
  
Member member2 = new Member();  
member2.setUsername("member2");  
member2.setHomeAddress(address);  
em.persist(member2);  
  
member.getHomeAddress().setCity("newCity");
```
위 예시에서는 같은 임베디드 타입을 두 Member에 주입했다.
```sql
update Member 
    set
        city=?,
        street=?,
        zipcode=?,
        USERNAME=?,
    where
        MEMBER_ID=?
update Member 
    set
        city=?,
        street=?,
        zipcode=?,
        USERNAME=?,
    where
        MEMBER_ID=?
```
update sql이 두 번 발생했다.
![](https://i.imgur.com/OX86sUe.png)
테이블에서 확인해보니 두 Member에 대한 임베디드 값의 필드 값이 모두 변경되었다.

그렇다면 어떻게 이러한 문제를 해결해야할까?

![](https://i.imgur.com/y3tVNtu.png)
값을 복사해서 사용해야 한다.
```java
Address address = new Address("city", "street", "10000");  
  
Member member = new Member();  
member.setUsername("member1");  
member.setHomeAddress(address);  
em.persist(member);  
  
Address copyAddress = new Address(address.getCity(), address.getStreet(), address.getZipcode());  
  
Member member2 = new Member();  
member2.setUsername("member2");  
member2.setHomeAddress(copyAddress);  
em.persist(member2);  
  
member.getHomeAddress().setCity("newCity");  
```
![](https://i.imgur.com/W1UdDP3.png)
`copyAddress`처럼 값을 복사할 경우 결과적으로 기대한 Member의 값만 변경되었음을 볼 수 있다.

하지만 참조 값을 직접 대입하는 것을 컴파일 에러 등으로 막을 방법이 없다.
따라서 값 타입은 항상 불변 객체로 설계해야 한다.
또 생성자로만 값을 설정하고 수정자를 사용하지 않으면 컴파일 레벨에서 막을 수 있다.
### 3️⃣ 컬렉션 값 타입
값 타입을 하나 이상 저장할 때 사용한다.
데이터베이스는 컬렉션을 같은 테이블에 저장할 수 없기 때문에 컬렉션을 저장하기 위한 별도의 테이블이 필요하다.
#### 3.1 사용
```java
@Entity  
public class Member extends BaseEntity {  
  
    @Id  
    @GeneratedValue  
    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
  
    @ElementCollection  
    @CollectionTable(name = "FAVORITE_FOODS", joinColumns = @JoinColumn(name = "MEMBER_ID"))  
    @Column(name = "FOOD_NAME")  
    private Set<String> favoriteFoods = new HashSet<>();  
  
    @ElementCollection  
    @CollectionTable(name = "ADDRESS", joinColumns = @JoinColumn(name = "MEMBER_ID"))  
    private List<Address> addressHistory = new ArrayList<>();
}
```
`@ElementCollection`, `@CollectionTable`을 사용해서 매핑한다.
별도의 테이블을 FK로 묶기 위해서 `joinColumns`로 설정한다.
`Set<String>`처럼 하나의 필드만을 가진 컬렉션 값 타입의 경우 `@Column`으로 컬럼명을 명시할 수 있다.
```sql
create table Member (
	MEMBER_ID bigint not null,
	USERNAME varchar(255),
	primary key (MEMBER_ID)
)
    
create table FAVORITE_FOODS (
	MEMBER_ID bigint not null,
	FOOD_NAME varchar(255)
)

create table ADDRESS (
	MEMBER_ID bigint not null,
	city varchar(255),
	street varchar(255),
	zipcode varchar(255)
)
```
테이블이 분리되어 생성되었다.

##### 1) 저장
```java
Member member = new Member();  
member.setUsername("member1");  
  
member.getFavoriteFoods().addAll(Arrays.asList("치킨", "족발", "피자"));  
member.getAddressHistory().addAll(Arrays.asList(new Address("old1", "street", "10000"), new Address("old2", "street", "10000")));
```
```sql
insert 
    into
        Member (USERNAME, MEMBER_ID) 
    values
        (?, ?)

insert 
    into
        ADDRESS (MEMBER_ID, city, street, zipcode) 
    values
        (?, ?, ?, ?)
insert 
    into
        ADDRESS (MEMBER_ID, city, street, zipcode) 
    values
        (?, ?, ?, ?)

insert 
    into
        FAVORITE_FOODS (MEMBER_ID, FOOD_NAME) 
    values
        (?, ?)
insert 
    into
        FAVORITE_FOODS (MEMBER_ID, FOOD_NAME) 
    values
        (?, ?)
insert 
    into
        FAVORITE_FOODS (MEMBER_ID, FOOD_NAME) 
    values
        (?, ?)
```
![](https://i.imgur.com/6Klz4bx.png)
저장 시 컬렉션 값 타입 각각을 persist 하지 않아도 생명 주기가 member에 의존되어 member만 영속화해도 값이 모두 저장된다.
##### 2) 조회
```java
Member findMember = em.find(Member.class, member.getId());
```
```sql
select
	m1_0.MEMBER_ID,
	m1_0.USERNAME,
from
	Member m1_0 
where
	m1_0.MEMBER_ID=?
```
조회 시에는 컬렉션 값 타입이 조회되는 sql이 발생하지 않았다.
```java
@Target({ElementType.METHOD, ElementType.FIELD})  
@Retention(RetentionPolicy.RUNTIME)  
public @interface ElementCollection {  
    Class targetClass() default void.class;  
  
    FetchType fetch() default FetchType.LAZY;  
}
```
`@ElementCollection` 어노테이션에서 그 답을 찾을 수 있다.
`FetchType.LAZY`로 설정되어 있으므로 컬렉션 값 타입은 지연 로딩 전략을 사용하기 때문에 Member 조회 시 함께 조회되지 않은 것이다.
##### 3) 수정
```java
findMember.getFavoriteFoods().remove("치킨");  
findMember.getFavoriteFoods().add("한식");
```
```sql
delete 
    from
        FAVORITE_FOODS 
    where
        MEMBER_ID=? 
        and FOOD_NAME=?

insert 
    into
        FAVORITE_FOODS (MEMBER_ID, FOOD_NAME) 
    values
        (?, ?)
```
Member를 통해서 조회한 컬렉션 값 타입을 변경했지만 수정(제거 및 저장)이 정상적으로 동작했다.

> 값 타입 컬렉션은 영속성 전이 (Cascade)  + 고아 객체 제거 기능을 필수로 가진다.

#### 3.2 주의
값 타입은 엔티티와 다르게 식별자 개념이 없다.
위의 예시에서 볼 수 있었듯이, PK를 설정하지 않았다.
그렇기 때문에 값 변경 시 추적이 어렵다.

컬렉션 값 타입에 변경사항이 발생하면, 주인 엔티티와 연관된 모든 데이터를 삭제하고, 컬렉션 값 타입에 있는 현재 값을 모두 다시 저장한다.
```java
findMember.getAddressHistory().remove(new Address("old1", "street", "10000"));  
findMember.getAddressHistory().add(new Address("newCity1", "street", "10000"));
```
`addressHistory`에 두 데이터가 저장되어 있는 상황에서 한가지를 지우고, 또 다른 데이터를 추가하는 코드를 작성했다.
```sql
/* one-shot delete for hellojpa.Member.addressHistory */
delete 
    from
        ADDRESS 
    where
        MEMBER_ID=?
insert 
    into
        ADDRESS (MEMBER_ID, city, street, zipcode) 
    values
        (?, ?, ?, ?)
insert 
    into
        ADDRESS (MEMBER_ID, city, street, zipcode) 
    values
        (?, ?, ?, ?)
```
주인 엔티티와 연관된 모든 데이터가 삭제되고, 다시 저장하는 sql이 발생한다.

따라서 컬렉션 값 타입 사용보다는 일대다 관계를 사용하는 것을 권장한다.
