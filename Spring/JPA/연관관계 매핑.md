## 1. 방향
### 1️⃣ 단방향
객체 연관관계는 참조(주소)로 맺는다. 
그렇기 때문에 참조용 필드가 있는 객체만 다른 객체를 참조하는 것이 가능하다.
그래서 단방향 연관관계를 가진다.

하지만 테이블 연관관계는 외래 키를 사용한다.
외래 키 하나로 양 쪽 테이블 JOIN이 가능하기 때문에 양방향 연관관계를 가진다.

```java
@Entity  
public class Member {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
  
    @ManyToOne  
    @JoinColumn(name = "TEAM_ID")  
    private Team team;
}

@Entity  
public class Team {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "TEAM_ID")  
    private Long id;  
  
    private String name;
}
```
위의 예시를 보면, Member만 Team의 참조를 가지고 있고, Team은 Member 참조를 가지지 않는다.
Member -> Team 단방향 연관관계가 맺어졌다는 것을 알 수 있다.

`@ManyToOne`은 다대일 관계라는 매핑 정보다.

`@JoinColumn`은 외래 키를 매핑할 때 사용한다. 
`name` 옵션에는 두 테이블이 연관관계를 맺고 있는 키를 지정한다. 
하지만, 이 어노테이션은 생략이 가능하다.
생략 시에는 기본 전략을 사용한다.
```
기본 전략 : 필드명 + _ + 참조하는 테이블의 컬럼명
```
### 2️⃣ 양방향
앞서 단방향에서 알아보았듯이, 테이블 연관관계는 이미 양방향 연관관계를 가지고 있다.
객체 연관관계를 양방향으로 만들기 위해서는 연관관계를 맺는 두 객체가 모두 서로의 참조 주소를 가지고 있으면 된다.
정확히 말하자면 양방향은 아니다. 단방향 연관관계 2개를 맺는 것이다.

```java
@Entity  
public class Team {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "TEAM_ID")  
    private Long id;  
  
    private String name;  
  
    @OneToMany(mappedBy = "team")  
    private List<Member> members = new ArrayList<>();
}
```
단방향 연관관계의 예시에서 Member 참조를 가지고 있지 않던 Team에도 참조 주소 필드를 만들었다.
이제 Team에서도 Member를 참조할 수 있으므로 양방향 연관관계를 가졌다고 할 수 있다.

`@OneToMany`는 일대다 관계라는 매핑 정보다.
#### 연관관계 주인
객체의 두 관계 중 외래 키를 관리하는 객체를 의미한다.
객체의 연관관계를 양방향으로 매핑하면서 하나의 관계에 참조 값이 두 개가 되었다.
결과적으로 `Member.team`으로도 관계를 정의할 수 있고, `Team.members`로도 관계를 정의할 수 있게 되었다.
하지만 DB의 입장에서 외래 키 하나만 알면 관리가 가능하고, 참조가 두개가 되면 오히려 원자성이 깨질 수도 있는 위험 요소가 발생한다.

그래서 연관관계의 주인을 두고 외래 키를 관리하게 되었다.
주인이 아니면 무언가에 의해서(by) 매핑이 되었다 (mapped)라는 의미를 가진 `mappedBy` 속성으로 주인을 지정해주어야 한다.

그렇다면 주인과 주인이 아닌 것의 차이는 무엇일까?

주인이 아닌 쪽은 외래 키를 읽기만 가능하다.
```java
Member member = new Member();  
member.setUsername("member1");  
  
Team team = new Team();  
team.setName("TeamA");  
team.getMembers().add(member);  
```
위의 예시에서 Team은 연관관계의 주인이 아니다. 현재 코드는 주인에 값을 입력하지 않고, 주인이 아닌 쪽에만 참조를 걸고 있다.
![](https://i.imgur.com/IwGzfU9.png)
주인이 아닌 쪽은 외래 키를 읽기만 가능하기 때문에 결과적으로 제대로 연관관계가 매핑되지 않게 되었다.

데이터베이스의 입장에서 옳은 연관관계 매핑은 아래와 같다.
```java
Team team = new Team();  
team.setName("TeamA");  
  
Member member = new Member();  
member.setUsername("member1");  
member.setTeam(team);  
```
연관관계의 주인에 값을 참조하는 것이다.

하지만, 객체지향적으로 옳은 연관관계 매핑은 아래와 같다.
```java
Team team = new Team();  
team.setName("TeamA");  
em.persist(team);  
  
Member member = new Member();  
member.setUsername("member1");  
member.setTeam(team);  
em.persist(member);  
  
team.getMembers().add(member);
```
양쪽의 참조에 모두 값을 넣어주는 것이다.

객체지향적으로 옳은 연관관계 매핑을 따라야 하는 이유는 JPA의 특성 때문이다.
```java
Team team = new Team();  
team.setName("TeamA");  
em.persist(team);  
  
Member member = new Member();  
member.setUsername("member1");  
member.setTeam(team);  
em.persist(member);  
  
Team findTeam = em.find(Team.class, team.getId());  
List<Member> members = findTeam.getMembers();  
for (Member m : members) {  
    System.out.println("m = " + m.getUsername());  
}
```
`em.persist()`로 Member와 Team 모두 1차 캐시에 영속화되었다.
이 상태에서 `em.find(Team.class, team.getId())` 호출을 하면 1차 캐시에서 바로 Team을 가져오기 때문에 select 쿼리가 나가지 않는다.
결국, 이렇게 가져온 Team의 참조값은 값이 저장되어 있지 않게 된다.
따라서 JPA가 커밋하기 전까지는 데이터의 원자성이 지켜지지 않는 것처럼 보인다.

그래서 양쪽의 참조값에 모두 값을 넣어주는 것이 바람직하다.

양쪽의 참조 객체를 모두 호출해 각각 넣어주는 것도 좋지만, 연관관계 편의 메소드를 생성할 경우 실수할 확률이 줄어든다.
```java
@Entity  
public class Member {  
  
    //... 
  
    public void changeTeam(Team team) {  
	    this.team = team;  
	    team.getMembers().add(this);  
	} 
}
```
이 경우, 연관관계 주인 쪽에서만 값을 저장하는 메서드를 호출하더라도 양쪽에 값이 설정된 결과를 낳는다.

> setter를 쓰지 않는 이유는 자바 관례상 존재하는 getter, setter로써의 호출이 아닌 양방향 연관 관계를 맺는다는 의미를 내포하기 위함이다.
## 2. 다중성
### 1️⃣ 다대일
가장 많이 사용하는 연관관계이다.
#### 1) 단방향
![](https://i.imgur.com/57QSoaQ.png)
```java
@Entity  
public class Member {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
    
    @ManyToOne  
    @JoinColumn(name = "TEAM_ID")  
    private Team team;
}

@Entity  
public class Team {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "TEAM_ID")  
    private Long id;  
  
    private String name;
}
```
회원은 `Member.team`으로 팀 엔티티를 참조할 수 있지만, 반대로 팀에는 회원을 참조하는 필드가 없다.

Member는 `@JoinColumn()`을 사용하여 `Member.team` 필드를 TEAM_ID 외래 키와 매핑한다. 
즉, `Member.team` 필드로 회원 테이블의 TEAM_ID 외래키를 관리하는 것이다.
#### 2) 양방향
![](https://i.imgur.com/JPkR98F.png)
```java
@Entity  
public class Member {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
    
    @ManyToOne  
    @JoinColumn(name = "TEAM_ID")  
    private Team team;
}

@Entity  
public class Team {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "TEAM_ID")  
    private Long id;  
  
    private String name;
    
    @OneToMany(mappedBy = "team")  
	private List<Member> members = new ArrayList<>();
}
```
객체로 봤을 때 Member, Team이 양쪽을 모두 참조하고 있지만 외래키는 다(N) 쪽인 Member만 가지고 있다.
즉, 연관관계의 주인은 Member.team이다.
### 2️⃣ 일대다
#### 1) 단방향
![](https://i.imgur.com/xb1zqSh.png)
보통 자신이 매핑한 테이블의 외래 키를 관리하지만, 테이블에서 외래키는 다(N) 쪽에 있기 때문에 `Team.members`로 Member 테이블의 TEAM_ID 외래 키를 관리한다.
```java
@Entity  
public class Member {  
  
    @Id  
    @GeneratedValue    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;
}

@Entity  
public class Team {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "TEAM_ID")  
    private Long id;  
  
    private String name;  
  
    @OneToMany  
    @JoinColumn(name = "TEAM_ID")  
    private List<Member> members = new ArrayList<>();
}
```
Member와 Team을 저장해보자.
```java
Member member = new Member();  
member.setUsername("member1");  
em.persist(member);  
  
Team team = new Team();  
team.setName("teamA");  
team.getMembers().add(member);  
em.persist(team);  
  
tx.commit();
```
```sql
Hibernate: 
    /* insert for
        hellojpa.Member */insert 
    into
        Member (USERNAME, MEMBER_ID) 
    values
        (?, ?)
Hibernate: 
    /* insert for
        hellojpa.Team */insert 
    into
        Team (name, TEAM_ID) 
    values
        (?, ?)
Hibernate: 
    update
        Member 
    set
        TEAM_ID=? 
    where
        MEMBER_ID=?
```
첫번째로 Member를 저장하는 Insert 쿼리가 발생했고, 두번째로 Team을 저장하는 Insert 쿼리가 발생했다.
하지만, 다대일 매핑과 다르게 Member의 TEAM_ID을 업데이트 하는 sql이 추가로 발생한다.
앞서 말했다시피 테이블에서는 외래키를 다(N) 쪽이 가지고 있기 때문에 Team 엔티티를 추가할 경우, Member 테이블에 접근해서 수정해야 한다는 의미가 된다.
그렇기 때문에 Team을 추가했지만 Member에 대한 update 쿼리가 추가적으로 발생하게 된다.

> ⚠️ 위와 같은 이유로 일대다 매핑보다는 다대일 매핑을 사용하는 것이 좋다.
#### 2) 양방향
![](https://i.imgur.com/SC3wJXr.png)
```java
@Entity  
public class Member {  
  
    @Id  
    @GeneratedValue    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
  
    @ManyToOne  
    @JoinColumn(name = "TEAM_ID", insertable = false, updatable = false)  
    private Team team;
}

@Entity  
public class Team {  
  
    @Id  
    @GeneratedValue    @Column(name = "TEAM_ID")  
    private Long id;  
  
    private String name;  
  
    @OneToMany  
    @JoinColumn(name = "TEAM_ID")  
    private List<Member> members = new ArrayList<>();
}
```
`@JoinColumn()`에 `insertable=false, updatable=false` 옵션을 두어 읽기 전용 필드로 설정해서 객체적으로만 양방향처럼 사용할 수 있도록 한다.

> ⚠️ JPA Spec 상 공식적으로 존재하는 매핑이 아니다. 다대일 양방향을 사용하는 것이 적절하다.
### 3️⃣ 일대일
주 테이블이나 대상 테이블 중에 외래 키 선택이 가능하다.
외래 키에 데이터베이스 유니크 (UNI) 제약 조건을 추가한다.
#### 1) 단방향
![](https://i.imgur.com/apSXyIc.png)
```java
@Entity  
public class Member {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
  
    @OneToOne  
    @JoinColumn(name = "LOCKER_ID")  
    private Locker locker;
}

@Entity  
public class Locker {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "LOCKER_ID")  
    private Long id;  
  
    private String name;  
}
```
---
만약, 대상 테이블에 외래 키가 있다면 어떻게 될까?
![](https://i.imgur.com/P0LbXrn.png)
이 경우 JPA가 지원하지 않아 설정이 불가능하다.
#### 2) 양방향
![](https://i.imgur.com/zkUA7KQ.png)
```java
@Entity  
public class Member {  
  
    @Id  
    @GeneratedValue    
    @Column(name = "MEMBER_ID")  
    private Long id;  
  
    @Column(name = "USERNAME")  
    private String username;  
  
    @OneToOne  
    @JoinColumn(name = "LOCKER_ID")  
    private Locker locker;
}

@Entity  
public class Locker {  
  
    @Id  
    @GeneratedValue    @Column(name = "LOCKER_ID")  
    private Long id;  
  
    private String name;  
  
    @OneToOne(mappedBy = "locker")  
    private Member member;  
}
```
---
![](https://i.imgur.com/GP1sT3F.png)
단방향일 때 대상 테이블에 외래 키가 있는 것은 불가능했지만, 양방향은 지원이 가능하다.
### 4️⃣ 다대다
객체는 컬렉션을 사용해서 객체 2개로 다대다 관계가 가능하다.
또한 `@ManyToMany`를 사용할 수 있어 편리해보이기도 하다.

![](https://i.imgur.com/1Tc56TC.png)
하지만 관계형 데이터베이스는 정규화된 테이블 2개로 다대다 관계를 표현할 수 없다.

`@ManyToMany`를 사용할 수 있지만, 다른 데이터를 필드로 추가할 수 없으며 쿼리로 찾아내기 어렵다는 단점이 있다.

따라서 연결 테이블을 엔티티로 승격해서 일대다, 다대일 관계로 풀어내야 한다.
